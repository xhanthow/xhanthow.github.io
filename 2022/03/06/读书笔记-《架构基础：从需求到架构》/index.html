读书笔记--《架构基础：从需求到架构》
2022-03-06| 架构 笔记|Word count: 865|Reading time: 2 min

推荐值：★★★★☆

花了几个晚上把这本书走读了一遍，由于之前也看过《大型网站技术架构：核心原理与案例分析》和《从 0 开始学架构》,感觉和他们讲得都差不多…

第一部分

“如何规划自己的架构师职业生涯”

第一部分围绕着“架构师”这个名词讲了what和how。由于自己最近也开始转型做设计，也是有点体会的：开发同学大多是与代码打交道；架构师大多是与人打交道，向上汇报、向下传达，且需要有大局观，而不是关注自己的一亩三分地。可以对着如下2个图来构建自己的架构能力：

可以对着上图进行查漏补缺！

第二部分

“大型互联网架构设计的四大原则”、“分布式微服务架构设计”

第二部分从宏观角度上讲述了“高可用”、“高伸缩性”、“高并发”和“安全”设计。

  • 高可用：主要思路就是冗余、副本、故障转移…对于不同层级服务（接入层、业务层、中间件等等）的设计都有自己的着重点！
  • 高伸缩性：垂直扩展、水平扩展–>尽量设计成无状态的服务
  • 高并发：多级缓存、异步化、读写分离
  • 安全：比较容易忽视的点，也是专业性比较强的点（受限于篇幅，也讲不出来东南西北）

然后围简单讲述了微服务架构（好像现在离开了微服务就没法活了🐕…）

老生常谈的套路：单体应用😔->SOA架构😊->ESB之殇😔->微服务架构😊

但是如果真的想要愉快的使用上微服务架构的话，其所需的服务治理能力一定要具备！！！

第三部分

第三部分主要围绕着各种各样的案例来展开的。记录下值得关注的点，一些细节的东西只有真正去做的时候才更有感悟！

  • 登录功能架构设计：
    • 网银的动态令牌：是靠着TOTP（Time-based One-Time Password，基于时间的一次性密码算法）
    • 本机号码一键登录：以来运营商的开放服务，授权页面也是运营商提供的
    • 多端会话：考虑到踢出功能，存储结构->userId+secretId+token
  • 用户安全架构设计：
    • 验证码：必须做到权限隔离，存储时需要增加验证码类型（适用的业务场景）
    • 修改密码：记录密码修改的轨迹信息
    • RBAC（Role Based Access Control，基于角色的访问控制）
    • AT和RT的玩法：这篇文章说得比较清楚Access Token & Refresh Token 详解以及使用原则
  • 系统攻防架构设计：
    • 验证码：不能将验证码以文字形式返回，必须是以图片返回
    • 防暴力穷举：虽然主键可以设计为具有顺序的数字，但是对外暴露时要形成新的编码
  • 系统消息架构设计：
    • 站内信存储设计：当一调消息产生时，只需要将消息插入消息定义表，不需要插入用户消息表，并不是每个用户都是活跃状态，当用户进入个人主页，查询未读消息数量时，只需要结合查询就可以获取到用户未读的消息数量
